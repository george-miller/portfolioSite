Clean Code

    Writing code has always been an enjoyable process, from the first program I wrote. Turning ideas into executable code and actually seeing that code work gives me a feeling of accomplishment and pride. However, it wasn't until I read the books Pragmatic Programming, Clean Code, and The Clean Coder that I realized how little I knew about the professional aspect of code. When I look back at the code I wrote as a beginner, I am astounded at how disorganized it was. Sure, I was able to get it to work, but as I read it now, I cannot even dissern what each function is supposed to do. Practices like meaningful naming, test driven development, and merciless refactoring have helped my code projects become readable, coherent pieces of professional software.
    The first time I finished a good refactoring session was an awesome feeling. Simply looking at the code I had written put a smile on my face; it was logical. The rationality of it was inevitable. The benefits of good planning and well thought out structure were enormous. I felt in control and I wasn't afraid to make a change here or try some new stuff over there. This is the feeling I strive for in any project I work on. After all of these experiences, I have learned that making the code clean is just as important as making the code work. If code that works isn't clean, it works as a result of coinsidence, not planning, and thus become a mess. The develpoer is afraid to touch the mess, because he really doesn't know how it works. How can we expect code to work if we don't know how it works ourselves? The answer is keep our disciplines at hand, and never tolerate a mess.

Logical Systems

    Code is a tool used to create systems and define their functionality. When creating any system, it is crucial to seperate the system's functionality into distinguishable parts and organize that functionality in a logical way. For example, the logical way a flashlight should be arranged is with the power button on the side and the circuitry inside. If I were making a flashlight, I wouldn't put the power button on the top because that is where the light goes. I also wouldn't put the circuitry that allows the flashlight to glow in a seperate container, where wires are connecting the flashlight to the container. I would put the circuitry into the flashlight, and I would put the power button on the side, so people can use it comfortably. Organizing the flashlight in this way also makes it easy for anyone to pick it up and use it with ease because it is arranged like one would expect.
    This same idea applies to systems created by programmers. If they are not arranged in a logical way, they will be harder to understand. When reading your code, colleages would have to dive deep into the implementation of the code, creating confusion and misunderstanding of the intent of your program. If code is arranged in a logical way, it is immensly easier to understand. If you have one class doing all the work for what really should be a set of objects, the inner workings of that class are going to become conveluted. The reader would have to go through the entire 2,000 line long class to figure out why it is not working as expected.
    If you have seperated your functionality into discrete and well-defined groups (aka modules) where code is organized based on subproblems, the reader can look at your class defenitions and get a big picture sense of what is going on and how it is being done. Then if they need to get into the implementation details, they always know what is being done and why it is being done. When extending a system of code such as the one I just described, a big picture understanding of the codebase is essential. It allows you to see how your piece of code interacts with other pieces of code in the system and how your functionality gets used.
    Without this understanding, bugs become huge roadblocks in your program because the programmer doesn't know what they mean or how to fix them. The only way to make sure the system you are creating is logical is to really take the time to understand the problem you are faced with, then plan a concrete and consistent outline for your system. After that you make tests to test the overall functionality and integration of the system, then you would make unit tests to test individual pieces of functionality. Once the unit tests pass, make the integration tests pass. At the end of it you will be left with an understandable, reuseable system.
